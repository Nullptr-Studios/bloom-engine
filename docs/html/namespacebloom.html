Detailed Description

Contains the core classes and functions for the bloom engine.

The bloom namespace encapsulates the main components and utilities required for the bloom engine. This includes the main engine class, window management, factory for object creation, and core utilities.

The bloom namespace is designed to provide a comprehensive set of tools for managing the game loop, rendering, and event handling. It includes classes for initializing and managing various subsystems such as the window, renderer, and input handling. The namespace is structured to facilitate the creation, management, and execution of a game using the bloom engine.

Key Classes:

  • Engine: Manages the main game loop, rendering, and event handling.
  • Window: Manages the window and its events.
  • Factory: Handles the creation and management of game objects.

Example Usage:

// Create a class that inherits from the Engine class (sandbox.hpp)
class Sandbox : public bloom::Engine {
public:
Sandbox() = default;
void OnBegin() override;
void Tick() override;
void End() const override;
};
// And then create an instance of the engine (sandbox.cpp) using the CreateEngine function
using namespace bloom;
std::unique_ptr<bloom::Engine> bloom::CreateEngine() { return std::make_unique<sandbox::Sandbox>(); }

enum bloom::EventCategory

Represents categories of events, allowing for logical grouping of event types.

Events are grouped into categories to facilitate efficient filtering and handling. Each category is represented by a single bit flag, enabling bitwise operations to determine if an event belongs to one or more categories.

Function Documentation

◆ CreateEngine()

std::unique_ptr< bloom::Engine > bloom::CreateEngine ( )

Creates an entry point for the Bloom engine.

This function serves as the primary entry point for the engine, allowing it to handle the main execution flow instead of the game directly. By abstracting the main function, the engine can take full control over initialization, execution, and shutdown processes.

Note
For future Xein: This design is intentional to ensure that the engine manages the main function rather than the game project. Any game-specific logic should integrate with this entry point.

While this function is declared here, its implementation resides in main.cpp within the game project. This separation ensures modularity between the engine and game layers.

Returns
A pointer to the Engine class instance, which is used to manage the engine's lifecycle.
